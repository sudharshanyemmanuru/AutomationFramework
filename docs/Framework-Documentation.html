1) Introduction

This framework is a keyword-driven Playwright automation framework where test steps are authored in Excel sheets and executed through reusable action handlers.

It is designed for both automation engineers and business users by separating test intent (what to do) from technical implementation (how to do).

2) Framework Capabilities

  • Data-driven transaction execution from TestSuite and ObjectRepository.
  • Supports common UI actions (input, click, validation, dropdown, window/frame switch).
  • Dynamic data read/write support for cross-step data management.
  • Download handling and advanced PDF comparison with HTML diff reports.
  • Config-driven behavior using .env (URLs, paths, report settings, exclusions).

3) Architecture Diagram

Legend: The framework takes data from Excel + .env, routes steps through execution layers, and generates actionable reports.

4) Execution Flow (Standard Procedure)

  1. Read test cases from TestSuite control sheet and filter executable rows.
  2. Open browser using APPLICATION_URL.
  3. Resolve each transaction to corresponding TestSuite and ObjectRepository sheets.
  4. For each step, map fieldType + action to the exact performer method.
  5. Execute UI/API/data actions; generate Playwright + custom reports for evidence.

10) Extensibility Guide for Automation Testers

This framework follows a dispatch pattern where each step is mapped by fieldType + action. To extend the framework safely, add code in a controlled sequence: ActionPerformer method → StepExecutor route → Excel updates.

A) Add New Action to Existing Field Type

Use case: Add action DBL_C (double click) for existing Button field type.

  1. Create a reusable method in ActionPerformer.js.
  2. Add a new route condition in StepExecutor.js.
  3. Update ObjectRepository row with field type and new action keyword.
  4. Add test data row in TestSuite transaction sheet.
/* ActionPerformer.js */ async performDoubleClickAction(testData, locator){ if(testData !== null){ await this.page.locator(locator).dblclick(); } } /* StepExecutor.js */ else if(objectRepositoryStep.fieldType==="Button" && objectRepositoryStep.action==="DBL_C"){ await this.actionPerformer.performDoubleClickAction(testSuiteStep.data, objectRepositoryStep.xpath); }
Technical recommendation: keep action methods idempotent where possible and always include clear guard clauses for null/invalid test data.

B) Add a Brand-New Field Type

Use case: Add FileUpload field type with action U.

  1. Define a strongly scoped method in ActionPerformer.js.
  2. Register new dispatch branch in StepExecutor.js.
  3. Add ObjectRepository row with FieldType=FileUpload, Action=U, locator for file input.
  4. Provide file path in TestSuite data cell.
/* ActionPerformer.js */ import fs from "fs"; async performFileUploadAction(testData, locator){ if(testData !== null){ if(!fs.existsSync(testData)){ throw new Error(`Upload file not found: ${testData}`); } await this.page.locator(locator).setInputFiles(testData); } } /* StepExecutor.js */ else if(objectRepositoryStep.fieldType==="FileUpload" && objectRepositoryStep.action==="U"){ await this.actionPerformer.performFileUploadAction(testSuiteStep.data, objectRepositoryStep.xpath); }

D) Engineering Checklist (Production-Grade)

  • Use deterministic action keywords and avoid overloaded semantics.
  • Validate preconditions (path existence, locator validity, required env variables).
  • Emit meaningful failures with step context (fieldName, fieldType, action).
  • Attach artifacts when behavior is complex (downloaded files, diff reports, screenshots).
  • Prefer reusable utility classes for cross-cutting logic (parsing, comparison, transformations).

11) Documentation & Usage Standards

  • Use consistent naming between TestSuite Field Name and ObjectRepository Field Name.
  • Use explicit action keywords and valid field types only.
  • Store expected PDFs in expected folder with stable naming convention.
  • For dynamic documents, use EXCLUDE_CONTENT DSL to avoid false mismatches.
  • Attach generated reports as part of execution evidence in Playwright HTML report.
